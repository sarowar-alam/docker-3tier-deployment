1 Containerization

What is Containerization?

Containerization is a method of packaging an application together with its runtime, libraries, dependencies, and configuration into a single self-contained unit called a container. Containers share the host operating system kernel — they do not carry their own OS — making them significantly lighter than Virtual Machines.

Core principle: Build once, run consistently anywhere — development, staging, and production environments are identical when using containers.

Why Containerization?

📦

Portability

Runs consistently on any host with a container runtime — any cloud, any OS.

🔒

Isolation

Each container has its own filesystem, network, and process space.

⚡

Speed

Containers start in milliseconds, not the minutes required for a VM boot.

♻️

Immutability

An image built once runs identically in every environment it is deployed to.

📐

Scalability

Individual services in a microservices architecture scale independently.

🚀

CI/CD Ready

Container images are version-controlled artifacts, ideal for automated pipelines.

2 Docker Architecture

What is Docker & Docker Architecture

Docker is an open-source platform for building, distributing, and running containerized applications. It follows a client-server architecture: the Docker Client sends commands to the Docker Daemon, which performs the actual work.

3 VMs vs Containers

Docker vs Virtual Machines

The fundamental difference is where isolation occurs. VMs virtualise hardware — each VM runs a full guest OS. Containers virtualise the operating system — they share the host kernel and isolate at the process level.

🖥 Virtual Machine

  • Boot time: 1–5 minutes
  • Image size: 1–50 GB (full OS)
  • Memory: 512 MB – 4 GB per VM
  • CPU overhead: 5–15% (hypervisor)
  • Density: 10–30 per host
  • Isolation: Hardware-level (stronger)
  • Portability: Hypervisor-dependent

🐳 Docker Container

  • Boot time: Under 1 second
  • Image size: 5–500 MB (app + libs)
  • Memory: 10–100 MB per container
  • CPU overhead: ~1–3% (near-native)
  • Density: Hundreds per host
  • Isolation: Kernel-level (shared OS)
  • Portability: Any Docker-compatible host

When to Use Each

ScenarioRecommended
Different OS requirement (e.g., Windows on Linux host)VM
Hard multi-tenant isolation (finance, regulated workloads)VM
Microservices, APIs, and web applicationsContainer
CI/CD build and test environmentsContainer
Kubernetes-orchestrated workloadsContainer
Development environment parity across teamsContainer
Legacy applications requiring full OS controlVM
Modern cloud-native approach: VMs and containers are used together. Kubernetes nodes run as VMs (EC2, GCE) which host hundreds of containers — combining OS-level isolation from VMs with the density and speed of containers.
4 Images vs Containers

Docker Images vs Containers

Analogy: An image is a template (like a class in OOP). A container is a running instance of that template (like an object). One image can produce many independent containers.

Docker Image

A Docker image is a read-only, immutable snapshot of a filesystem and its configuration. Images are built in layers — each Dockerfile instruction creates one cached, shareable, immutable layer.

Image Layers (read-only, bottom to top): Layer 4: COPY src/ ./src/ ← application source code Layer 3: RUN npm install ← cached if package.json unchanged Layer 2: WORKDIR /app ← working directory Layer 1: FROM node:18-alpine ← shared base layer

Docker Container

A container is a running instance of an image. Docker adds a thin writable layer on top of the read-only image layers. All runtime changes go into this layer. When the container is removed, this layer is discarded.

5 Commands

Docker Commands Reference

Image Commands

docker build -t name:tag .

Build an image from the Dockerfile in the current directory

docker images

List all locally available images

docker pull image:tag

Download an image from a registry

docker push repo/image:tag

Upload a local image to a registry

docker rmi image:tag

Remove a local image from disk

docker history image:tag

Show all layers of a built image

Container Lifecycle Commands

docker run -d -p 80:80 image

Create and start a container in detached mode

docker ps

List running containers

docker ps -a

List all containers including stopped

docker stop name

Gracefully stop a container (SIGTERM)

docker start name

Start a previously stopped container

docker restart name

Stop then start a container

docker rm name

Remove a stopped container

Debugging & Monitoring

docker logs -f name

Stream live stdout/stderr from a container

docker exec -it name sh

Open an interactive shell inside a running container

docker inspect name

Full container or image metadata in JSON

docker stats

Live CPU, memory, and network usage per container

Volumes & Networks

docker volume create name

Create a named persistent volume

docker volume ls

List all Docker volumes

docker network create net

Create an isolated bridge network

docker network ls

List all Docker networks

6 Dockerfile

Dockerfile Basics

A Dockerfile is a plain text file containing instructions for building a Docker image. Docker reads it top to bottom, executing each instruction to produce a layered, reproducible image.

Common Instructions

InstructionPurpose
FROMSets the base image. Every Dockerfile starts here.
WORKDIRSets the working directory for all subsequent instructions.
COPYCopies files from the build context into the image.
ADDLike COPY, but can also extract archives and fetch URLs.
RUNExecutes a shell command during build. Result becomes a new layer.
CMDDefault command when container starts. Overridable at runtime.
ENTRYPOINTFixed executable that always runs. Arguments can be appended.
ENVSets environment variables available at build time and runtime.
EXPOSEDocuments the port the application listens on (informational).
ARGBuild-time variable, passed with --build-arg. Not in final image.
VOLUMEDeclares a mount point for persistent external storage.
USERSets the user the container runs as. Use non-root in production.

Node.js Example

Dockerfile — Node.js
FROM node:18-alpine

WORKDIR /app

# Copy manifest first — enables layer caching for npm install
COPY package*.json ./
RUN npm install --omit=dev

COPY src/ ./src/

EXPOSE 3000

USER node

CMD ["node", "src/server.js"]

Python Example

Dockerfile — Python
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["python", "app.py"]

Nginx Static Frontend (Multi-Stage)

Dockerfile — Nginx (Multi-Stage Build)
# Stage 1 — Build
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Stage 2 — Serve (build tools excluded from final image)
FROM nginx:1.25-alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Dockerfile Best Practices

PracticeReason
Use a specific image tag (e.g., node:18-alpine)Avoid unexpected breaking changes from latest
Use minimal base images (-alpine, -slim)Reduce image size and attack surface
Copy dependency files before source codeMaximise layer cache efficiency
Chain RUN commands with &&Reduce the total number of layers
Use .dockerignoreExclude node_modules, .git, logs from build context
Run as a non-root userImprove container security posture
Use multi-stage builds for compiled appsKeep production images small and clean
